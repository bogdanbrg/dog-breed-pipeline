Step 1

Ingestion

A Cloud Function fetches all 628 dog breeds from the Dog API on a daily schedule (02:00 UTC via Cloud Scheduler). Each run writes a raw NDJSON file to Cloud Storage partitioned by date, then loads the same data into BigQuery's bronze layer.

Key design decision: ingested_date is injected in-memory during the BigQuery load step using a BytesIO buffer — no second file is written to GCS. This keeps Cloud Storage as a clean archive of the unmodified API response.

Write disposition: WRITE_APPEND — each daily run adds rows. Deduplication happens downstream in dbt using QUALIFY ROW_NUMBER().

def _load_to_bigquery(breeds, run_date):
    ndjson = "\n".join(
        json.dumps({**r, "ingested_date": run_date})
        for r in breeds
    )
    job_config = bigquery.LoadJobConfig(
        source_format=NEWLINE_DELIMITED_JSON,
        autodetect=True,
        write_disposition=WRITE_APPEND,
    )
    client.load_table_from_file(
        io.BytesIO(ndjson.encode("utf-8")),
        table_ref,
        job_config=job_config
    ).result()
Step 2

Data Exploration & Quality

Before building any transformations, the raw data was explored directly — both in BigQuery and by inspecting raw GCS files — to understand the data structure and identify quality issues.

Null fields at source

perfect_for and bred_for are null for all 628 breeds. Confirmed by reading raw GCS files directly — this is an API-level gap, not a pipeline bug.

Mixed height & weight formats

The API returns both "38-46" and "Male: 45-53; Female: 43-53" in the same column. Handled using REGEXP_EXTRACT_ALL to extract all numbers and take MIN/MAX.

Temperament case inconsistency

"loyal" and "Loyal" appeared as separate values, splitting the count. Fixed by applying LOWER() before splitting in mart_temperaments.

Temperament inflation bias

"intelligent" appears in ~86% of breeds — low signal for distinguishing breeds. Noted as a data quality observation rather than a filtering dimension.

Misleading breed names

Denmark Feist has country_code = US — "Denmark" refers to a town in Tennessee, not the country. A good example of why raw fields need validation before use in dashboards.

Step 3

Transformation with dbt

Transformations follow a medallion architecture — raw bronze data flows through a staging layer into curated mart tables ready for dashboards.

Bronze

dog_breeds_bronze

Raw JSON loaded directly from the Dog API. Append-only. Preserves full history.

Curated

dog_breeds_curated

dbt-transformed tables. Deduplicated, typed, enriched with derived fields.

stg_breeds

Deduplicates raw data (one row per breed), casts types, parses mixed-format range strings into min/max numeric columns. Materialised as a view.

dim_breed

Descriptive attributes per breed. Derives size_class (Toy / Small / Medium / Large / Giant) from weight_kg_max. Materialised as a table.

fact_weight_life_span

Numeric measurements per breed — weight, height, life span (min, max, avg). Pure fact table with no descriptive columns. Materialised as a table.

mart_breeds

Denormalised join of dim + fact. Dashboard-ready, one row per breed, all columns in one place. Materialised as a table.

mart_temperaments

Bridge table — splits the comma-separated temperament string into one row per breed per tag. Powers the temperament frequency chart. Materialised as a table.

Step 4

Testing & Documentation

dbt tests ensure data quality is enforced on every run — both in development (CI on pull requests) and production (CD on merge to main).

Generic tests: unique, not_null, accepted_values, and relationships (referential integrity between fact and dim).

Custom tests (4):

  • assert_life_span_min_lte_max — returns any row where life_span_min > life_span_max, catching data anomalies from the API range parser
  • assert_row_count_in_range — stg_breeds must have 600–700 rows, catching partial or failed ingestion
  • assert_life_span_max_realistic — life_span_max must be between 1–30 years
  • assert_breed_group_null_rate — null rate for breed_group must stay below 50%

Documentation: All models and columns have descriptions in schema.yml, persisted to BigQuery via persist_docs.

-- tests/assert_life_span_min_lte_max.sql
-- Returns rows where min > max (should be 0)
select
    breed_id,
    life_span_min,
    life_span_max
from {{ ref('stg_breeds') }}
where life_span_min > life_span_max

Step 4b

Data Quality & Observability

Beyond generic dbt tests, three custom data quality tests were added to catch source-level issues that generic tests cannot detect.

Row count validation — asserts that stg_breeds always has between 600-700 rows. Catches incomplete API responses or deduplication failures.

Value range check — asserts that life_span_max is always between 1-30 years. Catches parsing errors from the REGEXP_EXTRACT_ALL logic.

Null rate monitoring — asserts that no more than 50% of breeds have a null breed_group. Catches structural changes in the API response.

Cloud Monitoring alert — a log-based alerting policy sends an email to bogdanbirgovan@gmail.com whenever the Cloud Function logs an ERROR severity message. Pipeline failures are caught immediately rather than discovered when the dashboard goes stale.

-- assert_row_count_in_range.sql
select count(*) as row_count
from {{ ref('stg_breeds') }}
having count(*) < 600 or count(*) > 700

-- assert_life_span_max_realistic.sql
select breed_id, life_span_max
from {{ ref('stg_breeds') }}
where life_span_max is not null
  and (life_span_max < 1 or life_span_max > 30)

-- assert_breed_group_null_rate.sql
select
    countif(breed_group is null) / count(*) as null_rate
from {{ ref('stg_breeds') }}
having null_rate > 0.50
Step 5

CI/CD with GitHub Actions

Two workflows automate quality checks and deployment. Authentication to GCP uses Workload Identity Federation — no service account keys stored in GitHub secrets.

CI — Pull Request

ci.yml

Triggers on every PR to main. Runs dbt run and dbt test against the dev dataset. Blocks merge if any test fails.

CD — Merge to Main

cd.yml

Triggers on push to main. Deploys the Cloud Function, then runs dbt run and dbt test against prod. Full deploy in under 3 minutes.

Step 5b

Orchestration & Scheduling

The ingestion function runs daily at 02:00 UTC via Cloud Scheduler. dbt currently runs on every deploy via the CD workflow — which is sufficient for this dataset since breed data changes rarely.

In a production pipeline where source data changes daily, dbt should also run on a schedule — one hour after ingestion completes — to keep curated tables fresh.

This can be added to the existing GitHub Actions workflow with a simple schedule trigger. For more complex pipelines with task dependencies, a dedicated orchestrator like Cloud Composer (Airflow) or Prefect would be used instead.

# Add to cd.yml to schedule dbt daily
on:
  push:
    branches: [main]
  schedule:
    # Ingest runs at 02:00 UTC via Cloud Scheduler
    # dbt runs at 03:00 UTC — after ingestion completes
    - cron: '0 3 * * *'
